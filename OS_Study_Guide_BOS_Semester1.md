## Тема 1. Компиляция, сборка, запуск

### Глоссарий

| Термин | Краткое определение |
|--------|---------------------|
| **Препроцессор** | Первая фаза компиляции: раскрывает `#include`, `#define`, макросы |
| **Компилятор** | Переводит исходный C-код в объектный файл (`.o`) на ассемблере/машинном коде |
| **Компоновщик (linker)** | Объединяет `.o`-файлы и библиотеки в исполняемый файл |
| **Объектный файл** | Промежуточный результат компиляции; содержит машинный код одного `.c`-файла |
| **Unresolved symbol** | Символ (функция/переменная), объявленный, но ещё не привязанный к адресу |
| **Static library** | Архив `.a`: код копируется в исполняемый файл при компоновке |
| **Shared library** | Динамическая библиотека `.so`: загружается в память при запуске (или во время выполнения) |
| **`nm`** | Утилита: показывает таблицу символов объектного файла / исполняемого |
| **`ldd`** | Утилита: выводит зависимости исполняемого файла от разделяемых библиотек |
| **`dlopen` / `dlsym`** | POSIX API для загрузки `.so` во время выполнения программы |
| **PLT / GOT** | Таблицы для ленивой привязки символов в динамических библиотеках |
| **ELF** | Executable and Linkable Format — формат исполняемых файлов и библиотек в Linux |

### Подробное объяснение

Компиляция в C проходит четыре фазы: **препроцессор → компилятор → ассемблер → компоновщик**.
На выходе ассемблера получается объектный файл, который содержит таблицу символов; символы, определённые в других единицах трансляции, помечаются как *unresolved* (тип `U` в выводе `nm`).
Компоновщик разрешает эти символы: при **статической** компоновке он физически копирует код из `.a`-архива в исполняемый файл, поэтому `nm` покажет символ функции уже как определённый (тип `T`) — никакой зависимости от внешней библиотеки не остаётся.
При **динамической** компоновке (`.so`) исполняемый файл содержит только *stub* — заглушку в PLT (Procedure Linkage Table); реальный адрес функции записывается в GOT (Global Offset Table) динамическим компоновщиком (`ld.so`) при первом вызове.
`ldd` показывает, какие `.so`-файлы нужны программе и откуда они берутся; если библиотека не найдена — программа не запустится.
**Загрузка `.so` во время выполнения** через `dlopen(path, RTLD_LAZY)` / `dlsym(handle, "func_name")` позволяет получить указатель на функцию и вызвать её без каких-либо объявлений на этапе компоновки — это основа плагинных систем.
Переменная среды `LD_LIBRARY_PATH` расширяет пути поиска `.so`; `LD_PRELOAD` позволяет подменить любой символ ещё до старта программы.
Флаг `-fPIC` (Position Independent Code) обязателен для кода, который войдёт в `.so`: все адреса внутри него вычисляются относительно инструкции, а не абсолютно.
Сегменты ELF-файла (`.text`, `.data`, `.bss`, `.rodata`) определяют, как содержимое объектного файла размещается в адресном пространстве процесса.
Утилита `readelf -S` позволяет увидеть все секции, а `readelf -d` — динамическую секцию с зависимостями; это полезно при диагностике проблем с библиотеками.
Понимание этой темы критично для отладки ошибок `undefined reference`, `cannot open shared object file` и конфликтов символов при смешанной компоновке.

### Вопросы и ответы

**В: Чем отличается статическая библиотека от динамической?**
О: Статическая (`.a`) копирует код внутрь исполняемого файла при компоновке; динамическая (`.so`) остаётся отдельным файлом и загружается в память при запуске (или через `dlopen`). Статическая увеличивает размер бинарника, но не требует библиотеки на машине пользователя; динамическая экономит память (один экземпляр `.so` разделяется между всеми процессами) и позволяет обновлять библиотеку без перекомпиляции программы.

**В: Почему `nm` показывает функцию из статической библиотеки как resolved, а из динамической — нет?**
О: Из статической библиотеки код уже скопирован в `.o`-файл исполняемого, поэтому символ получает адрес и помечается `T` (text). Из динамической библиотеки в исполняемом файле есть только PLT-заглушка — реальный адрес неизвестен до загрузки `.so`; `nm` покажет символ как `U` (undefined) или `i` (PLT).

**В: Что такое PLT и GOT и зачем они нужны?**
О: PLT (Procedure Linkage Table) — таблица маленьких заглушек, по одной на каждый внешний символ. При первом вызове заглушка обращается к динамическому компоновщику, который находит реальный адрес функции и записывает его в GOT (Global Offset Table). При последующих вызовах PLT-заглушка сразу читает адрес из GOT — это называется **ленивой привязкой** и ускоряет старт программы.

**В: Как запустить программу, если `.so` лежит в нестандартном каталоге?**
О: Варианты: (1) добавить путь в `LD_LIBRARY_PATH`; (2) вшить путь в ELF с помощью флага `-rpath` при компоновке; (3) добавить путь в `/etc/ld.so.conf` и запустить `ldconfig`.

## Тема 2. Системные вызовы

### Глоссарий

| Термин | Краткое определение |
|--------|---------------------|
| **Системный вызов (syscall)** | Запрос программы к ядру ОС на выполнение привилегированной операции |
| **Пространство пользователя / ядра** | Режимы CPU (ring 3 / ring 0); syscall переключает между ними |
| **`strace`** | Утилита: перехватывает и выводит все системные вызовы процесса |
| **`write(2)`** | Syscall: записывает байты из буфера в файловый дескриптор |
| **`read(2)`** | Syscall: читает байты из файлового дескриптора в буфер |
| **`syscall()`** | Функция libc: вызывает произвольный syscall по номеру (обёртка над `syscall` инструкцией) |
| **Trap / INT 0x80 / `syscall` инструкция** | Механизм CPU для перехода из user space в kernel space |
| **`ptrace(2)`** | Syscall: позволяет одному процессу отслеживать и контролировать другой |
| **ABI (Application Binary Interface)** | Соглашение о том, в каких регистрах передаются аргументы и номер syscall |
| **`vDSO`** | Virtual Dynamic Shared Object: ядро отображает страницы с быстрыми псевдо-syscall (напр. `gettimeofday`) в адресное пространство процесса |

### Подробное объяснение

Системный вызов — единственный легальный способ для программы попросить ядро сделать то, что недоступно в user space: открыть файл, выделить память, отправить пакет, создать процесс.
При выполнении инструкции `syscall` (x86-64) или `svc` (ARM) процессор переключается в режим ядра (ring 0), сохраняет контекст пользовательского потока и передаёт управление обработчику в ядре.
Номер вызова передаётся в регистре `rax` (x86-64); аргументы — в `rdi, rsi, rdx, r10, r8, r9`; возвращаемое значение — снова в `rax` (отрицательное значение означает ошибку, его модуль — код `errno`).
`printf()` — функция libc, которая форматирует строку и в итоге вызывает `write(1, buf, len)` — тот самый системный вызов с дескриптором `1` (stdout).
`strace` использует `ptrace` для перехвата каждого syscall: он видит аргументы *до* исполнения и возвращаемое значение *после*; `strace -c` даёт статистику (число вызовов, время).
Функция `syscall(SYS_write, 1, "hello\n", 6)` из libc самостоятельно формирует вызов, не проходя через `printf` — это позволяет вызвать любой syscall без обёртки.
Для вызова syscall без libc вообще достаточно нескольких строк ассемблера: положить номер в `rax`, аргументы в нужные регистры, исполнить `syscall`.
`ptrace(PTRACE_TRACEME)` в дочернем процессе + `waitpid` в родителе + `PTRACE_GETREGS` позволяют реализовать собственный мини-`strace`.
Количество системных вызовов при запуске даже простой программы велико, потому что динамический компоновщик (`ld.so`) читает конфиг, загружает зависимые `.so`, настраивает `PLT/GOT` — всё это перед `main()`.
Понимание syscall-интерфейса критично для системного программирования, безопасности (seccomp-фильтры блокируют syscall по номеру) и написания эксплойтов / защит.

### Вопросы и ответы

**В: Как работает переключение из user space в kernel space при syscall?**
О: Программа помещает номер syscall в регистр `rax`, аргументы в остальные регистры и выполняет инструкцию `syscall`. Процессор сохраняет `rip`, `rsp` и флаги, переключает привилегии на ring 0, устанавливает `rip` на адрес обработчика из `MSR_LSTAR`. Ядро исполняет запрошенную операцию и возвращается в user space через `sysret`.

**В: Почему при запуске `hello world` `strace` показывает десятки вызовов, а не один `write`?**
О: Динамический компоновщик `ld.so` вызывается ядром первым (`execve` → `ld.so`). Он выполняет: `openat`/`read`/`mmap` для каждой `.so`-зависимости, `mprotect` для настройки прав страниц, `close` и т.д. Только после этого управление доходит до `main()` и одного `write`.

**В: Чем `syscall()` из libc отличается от написания ассемблерной вставки?**
О: Функционально они делают одно и то же — выполняют инструкцию `syscall`. Разница в удобстве: `syscall()` — переносимая C-функция, избавляет от ручного управления регистрами. Ассемблерная вставка даёт полный контроль и нулевые накладные расходы, но непереносима между архитектурами.

**В: Что такое `ptrace` и как он используется в отладчиках?**
О: `ptrace` позволяет одному процессу (трассировщику) получать уведомления о каждом syscall или сигнале другого процесса, читать/писать его память и регистры. `gdb` использует `ptrace(PTRACE_ATTACH)` для присоединения, `PTRACE_PEEKTEXT`/`PTRACE_POKETEXT` для чтения/записи кода и `PTRACE_SINGLESTEP` для пошагового исполнения.

## Тема 3. Файлы и файловые системы

### Глоссарий

| Термин | Краткое определение |
|--------|---------------------|
| **inode** | Структура ядра: хранит метаданные файла (тип, права, владелец, размер, указатели на блоки данных); не хранит имя |
| **dentry** | Directory Entry: кэш-объект ядра, связывающий имя (строку) с inode; ускоряет поиск путей |
| **file (struct file)** | Объект ядра, представляющий **открытый** файловый дескриптор; хранит смещение (`offset`) и права открытия |
| **Файловый дескриптор (fd)** | Целое число в таблице дескрипторов процесса; указывает на `struct file` |
| **Жёсткая ссылка (hard link)** | Дополнительная запись в директории, указывающая на тот же inode |
| **Символьная ссылка (symlink)** | Файл-указатель, хранящий путь к целевому файлу; может быть dangling |
| **VFS (Virtual File System)** | Уровень абстракции в ядре: унифицирует ext4, tmpfs, procfs и др. |
| **`stat(2)`** | Syscall: возвращает метаданные файла (из inode) |
| **`chmod(2)` / `chown(2)`** | Syscall'ы изменения прав и владельца |
| **`readdir` / `getdents`** | Syscall для перечисления содержимого директории |
| **`/proc/pid/pagemap`** | Файл procfs: маппинг виртуальных страниц процесса на физические фреймы |

### Подробное объяснение

В Linux **всё — файл**: обычные файлы, директории, сокеты, устройства, символьные ссылки — все они представлены через единый интерфейс VFS.
**inode** — сердце файловой системы: он хранит тип файла, права (`mode`), UID/GID, три временных метки (atime/mtime/ctime), размер, счётчик жёстких ссылок и указатели на блоки данных на диске. Имя файла **не** хранится в inode; оно живёт в директории (в записи dentry).
**Жёсткая ссылка** — это просто ещё одна запись в директории, ссылающаяся на тот же inode; файл существует, пока счётчик ссылок (`nlink`) не упадёт до нуля. Жёсткие ссылки не могут пересекать границы файловых систем и указывать на директории (за исключением специальных случаев `.` и `..`).
**Символьная ссылка** — отдельный inode типа `S_IFLNK`, содержащий путь к цели. Если цель удалена, ссылка становится «висячей» (dangling symlink). `readlink` читает этот путь, `stat` идёт по ссылке до конца, `lstat` возвращает метаданные самой ссылки.
**`struct file`** в ядре создаётся при `open(2)`: хранит текущее смещение (`pos`), флаги открытия и указатель на inode. Два процесса могут иметь разные `struct file` на один inode (разные смещения), и это нормально.
**`dentry`-кэш** (dcache) позволяет ядру не проходить весь путь по диску каждый раз: разбирает путь типа `/usr/bin/ls` на компоненты и кэширует каждый переход. При удалении файла его dentry помечается unhashed, inode освобождается только когда `nlink == 0` **и** все `struct file` закрыты.
**Права доступа** хранятся в inode как 12 бит: `rwxrwxrwx` + SUID/SGID/sticky. Ядро проверяет: если процесс — владелец, применяет биты owner; если в группе — group; иначе — other. `root` (UID 0) проверки прав не проходит (кроме execute-бита).
**`/proc/pid/pagemap`** — двоичный файл: на каждые 8 байт приходится одна виртуальная страница. Биты 0–54 — номер физического фрейма (PFN), бит 63 — страница присутствует в памяти, бит 62 — страница в swap. Это позволяет вычислить физический адрес из виртуального.
Функции для работы с файловой системой: `open/close/read/write` — работа с данными; `stat/lstat/fstat` — метаданные; `mkdir/rmdir/unlink/rename` — структура; `opendir/readdir/closedir` — перечисление директорий.
Знание разницы между inode, dentry и `struct file` часто спрашивают на системных собеседованиях — это показывает понимание того, что происходит внутри ядра при обычных файловых операциях.

### Вопросы и ответы

**В: В чём разница между жёсткой и символьной ссылкой?**
О: Жёсткая ссылка — дополнительное имя для того же inode; файл живёт, пока есть хоть одна ссылка; не пересекает файловые системы. Символьная ссылка — отдельный файл, содержащий путь; может быть dangling; работает между ФС и с директориями.

**В: Что произойдёт, если открыть файл, а потом удалить его (unlink) пока он открыт?**
О: `unlink` уменьшит `nlink` inode до 0 и удалит dentry, но `struct file` в ядре всё ещё держит ссылку на inode. Процесс продолжит читать/писать файл через дескриптор. Данные на диске освободятся только при `close` последнего дескриптора — это стандартный трюк для создания временных файлов без имени.

**В: Чем отличается `stat` от `lstat`?**
О: `stat` проходит по символическим ссылкам и возвращает метаданные **целевого** файла. `lstat` возвращает метаданные **самой ссылки** (тип `S_IFLNK`, размер — длина пути).

**В: Как ядро использует dentry-кэш при открытии файла?**
О: При `open("/usr/bin/ls", ...)` VFS разбивает путь на компоненты и для каждого ищет dentry в кэше. Если dentry найден — inode берётся из кэша, диск не читается. Промах кэша (`cache miss`) ведёт к чтению директории с диска и созданию нового dentry. Именно поэтому повторные открытия тех же файлов значительно быстрее.

## Тема 4. Адресное пространство процесса

### Глоссарий

| Термин | Краткое определение |
|--------|---------------------|
| **Виртуальная память** | Абстракция: каждый процесс видит свои непрерывные адреса, MMU переводит их в физические |
| **MMU (Memory Management Unit)** | Аппаратный блок CPU: транслирует виртуальные адреса в физические с помощью page table |
| **Страница (page)** | Минимальная единица управления памятью, обычно 4096 байт |
| **`/proc/pid/maps`** | Карта виртуального адресного пространства процесса: регионы, права, источник |
| **Сегменты ELF** | `.text` (код), `.data` (инициализированные глобалы), `.bss` (неинициализированные), `.rodata` (константы), heap, stack |
| **`mmap(2)`** | Syscall: присоединяет регион виртуальных адресов (к файлу или анонимный) |
| **`munmap(2)`** | Syscall: отсоединяет регион |
| **`mprotect(2)`** | Syscall: изменяет права доступа к региону (`PROT_READ`, `PROT_WRITE`, `PROT_EXEC`) |
| **Segmentation Fault (SIGSEGV)** | Сигнал от ядра при нарушении прав доступа к памяти (обращение к незамапленному региону или нарушение `mprotect`) |
| **COW (Copy-On-Write)** | Оптимизация `fork`: дочерний процесс разделяет страницы с родительским до момента записи |
| **Stack / Heap** | Стек — автоматическая память функций (растёт вниз); куча — динамическая (`malloc`/`free`), растёт вверх |
| **`brk/sbrk`** | Устаревшие syscall'ы для управления верхней границей heap; `malloc` обычно использует `mmap` |

### Подробное объяснение

Каждый процесс в Linux работает в **изолированном виртуальном адресном пространстве** размером (на 64-бит) до 128 ТБ для user space. Ядро делит его на регионы: код (`.text`), инициализированные данные (`.data`), неинициализированные данные (`.bss`), куча (`heap`), отображения библиотек, стек (`stack`) и специальный регион `vDSO`.
**Локальные переменные** функций живут на стеке — в текущем фрейме. Адрес локальной переменной становится недействительным сразу после возврата функции: фрейм уничтожается, но память физически не очищается — туда будет записан следующий фрейм. Возврат указателя на локальную переменную — **неопределённое поведение**.
**Глобальные инициализированные** переменные — в `.data`, **глобальные неинициализированные** — в `.bss` (область, заполняемая нулями ядром). **Константы** (`const` в глобальной области и строковые литералы) — в `.rodata`, которая маппируется только для чтения (`PROT_READ`). Попытка записи в `.rodata` → SIGSEGV.
**`malloc`** на современных реализациях использует `mmap(PROT_READ|PROT_WRITE, MAP_ANONYMOUS)` для больших аллокаций и `brk` для маленьких. После `free` память не возвращается ОС немедленно: аллокатор хранит освобождённые блоки в своих структурах для повторного использования. Именно поэтому обращение к указателю после `free` (use-after-free) может «работать», но является неопределённым поведением.
**`mmap`** — универсальный инструмент управления адресным пространством: можно отобразить файл в память (`MAP_SHARED` — изменения идут в файл, `MAP_PRIVATE` — COW), создать анонимный регион (`MAP_ANONYMOUS`) для кучи или стека.
**`mprotect`** позволяет изменить права доступа к региону. Запрет `PROT_READ` → чтение вызывает SIGSEGV; запрет `PROT_WRITE` → запись вызывает SIGSEGV. На этом механизме основаны защиты: non-executable stack (`NX`-бит), RELRO (GOT — только чтение после инициализации).
**Segmentation Fault** — это сигнал `SIGSEGV`, который ядро посылает процессу при page fault, который не может быть разрешён: обращение за пределы замапленного региона или нарушение прав `mprotect`. По умолчанию обработчик SIGSEGV завершает процесс с дампом.
**Переменные окружения** хранятся в самом верхнем регионе стека, рядом с аргументами командной строки (`argv`). Изменение переменной окружения через `setenv`/`putenv` меняет значение **только внутри процесса** — дочерние процессы, запущенные после этого, унаследуют новое значение, но родительская shell не изменится.
**`/proc/pid/maps`** — главный инструмент наблюдения за адресным пространством: для каждого региона показывает адреса, права (`rwxp`), смещение, устройство и имя файла-источника (или `[heap]`, `[stack]`, `[vdso]`).
Реализация собственной кучи поверх `mmap(MAP_ANONYMOUS)` требует хранить в самом регионе метаданные о свободных/занятых блоках (например, заголовок размера перед каждым блоком) — это суть `malloc`.

### Вопросы и ответы

**В: Что произойдёт, если вернуть указатель на локальную переменную из функции?**
О: Формально — неопределённое поведение. Практически — адрес остаётся в пределах стека, но при следующем вызове любой функции её фрейм перепишет эту память. Чтение по такому указателю может вернуть мусор или корректные данные (если стек ещё не использовался) — непредсказуемо.

**В: Чем `MAP_SHARED` отличается от `MAP_PRIVATE` в `mmap`?**
О: `MAP_SHARED` — запись в регион немедленно отражается в файле и видна другим процессам, маппирующим тот же файл. `MAP_PRIVATE` — при первой записи создаётся приватная копия страницы (COW); изменения не уходят в файл и невидимы другим.

**В: Что такое Segmentation Fault и как его вызвать намеренно?**
О: SIGSEGV — сигнал ядра при обращении к невалидному адресу или нарушении прав страницы. Намеренно: разыменовать `NULL` (`*(int*)0 = 1`), выйти за пределы массива на стеке, писать в `mmap(..., PROT_READ)` регион, или обратиться к области после `munmap`.

**В: Почему обращение к памяти после `free` иногда «работает»?**
О: Освобождённый блок остаётся в адресном пространстве процесса — память не отдаётся ОС, а помечается свободной в аллокаторе. Пока этот блок не переиспользован следующим `malloc`, данные там прежние. Это **use-after-free** — уязвимость, эксплуатируемая для hijack потока исполнения.

## Тема 5. Создание и завершение процесса

### Глоссарий

| Термин | Краткое определение |
|--------|---------------------|
| **`fork(2)`** | Syscall: создаёт точную копию текущего процесса; родитель получает PID дочернего, дочерний — 0 |
| **`exec(2)` (execve и семья)** | Syscall: заменяет образ процесса новой программой; PID не меняется |
| **`wait(2)` / `waitpid(2)`** | Syscall: ждёт завершения дочернего процесса и получает его статус |
| **`exit(2)` / `_exit(2)`** | Syscall: завершает процесс с кодом возврата |
| **Зомби-процесс (zombie)** | Процесс завершился, но родитель не вызвал `wait` — запись в таблице процессов ещё существует |
| **Сирота (orphan)** | Процесс, чей родитель завершился раньше него; усыновляется `init`/`systemd` (PID 1) |
| **`clone(2)`** | Низкоуровневый syscall для создания процессов/потоков с гибкими настройками разделения ресурсов |
| **COW (Copy-On-Write)** | Механизм оптимизации `fork`: страницы разделяются до первой записи |
| **PID / PPID** | Process ID / Parent Process ID |
| **`/proc/pid/status`** | Файл procfs с состоянием процесса (R/S/Z/D/T) |
| **Код завершения (exit code)** | 8-битное число (0–255), передаваемое родителю через `wait` |

### Подробное объяснение

**`fork()`** создаёт дочерний процесс как копию родительского: одинаковые сегменты памяти, файловые дескрипторы, сигнальные маски. Благодаря **COW** страницы реально не копируются — они помечаются read-only и копируются только при записи. Дочерний процесс получает `fork() == 0`, родительский — PID дочернего. Переменные имеют одинаковые виртуальные адреса, но после первой записи дочернего — разные физические страницы.
Дочерний и родительский процессы исполняются независимо после `fork`. Изменение переменной в дочернем не влияет на родителя — это фундаментальное свойство изоляции процессов. Порядок исполнения родителя и дочернего не определён — нужна синхронизация (`wait`, сигналы, pipes).
**`exec`** заменяет весь образ процесса (код, стек, heap) новой программой, PID остаётся прежним. Файловые дескрипторы наследуются (кроме тех, у которых установлен `FD_CLOEXEC`). Именно `fork + exec` — стандартная идиома запуска новой программы.
**Зомби** появляется когда дочерний процесс завершился (`exit`), но родитель не вызвал `wait`. Ядро должно хранить запись в таблице процессов (с кодом завершения), пока родитель её не заберёт. Зомби не потребляет CPU или память (кроме строчки в таблице), но если родитель никогда не вызывает `wait` — таблица процессов может переполниться.
Если **родитель завершился раньше дочернего**, дочерний становится **сиротой** и усыновляется PID 1 (`systemd`/`init`). `systemd` вызывает `wait` за сирот автоматически.
**`clone(2)`** — это тот syscall, который внутри вызывают и `fork`, и `pthread_create`. С флагами `CLONE_VM | CLONE_FILES | CLONE_FS | ...` создаётся **поток** (разделяет адресное пространство); без этих флагов — полноценный процесс. Стек для нового процесса/потока нужно выделить вручную (обычно через `mmap`).
Состояния процесса: **R** (Running/Runnable), **S** (Sleeping — interruptible), **D** (Sleeping — uninterruptible, обычно I/O), **Z** (Zombie), **T** (Stopped). Видны в `/proc/pid/status` (поле `State:`) или `ps aux`.
**`waitpid(pid, &status, options)`** возвращает статус через макросы: `WIFEXITED(status)` — нормальное завершение, `WEXITSTATUS(status)` — код; `WIFSIGNALED(status)` — убит сигналом, `WTERMSIG(status)` — номер сигнала. Кода завершения не будет, если процесс убит сигналом.
Исследование `/proc/pid/maps` до и после `fork` показывает одинаковые виртуальные карты; после `exec` — полностью новую карту. Это наглядно демонстрирует семантику обоих syscall'ов.

### Вопросы и ответы

**В: Что такое зомби-процесс и почему он существует?**
О: Зомби — завершившийся процесс, чья запись в таблице процессов ещё не удалена, потому что родитель не вызвал `wait`. Запись нужна, чтобы родитель мог узнать код завершения. Зомби не потребляет ресурсы, кроме строчки в таблице. Лечение: исправить родителя чтобы вызывал `waitpid`, или завершить родителя (тогда `init` заберёт зомби).

**В: Что происходит с памятью при `fork`? Почему это эффективно?**
О: Ядро не копирует физические страницы — помечает их read-only и разделяет между родителем и дочерним (COW). При первой записи в любую страницу возникает page fault, ядро копирует страницу и даёт каждому свою копию. Это делает `fork` быстрым даже для больших процессов — чаще всего за ним сразу следует `exec`, и копировать ничего не придётся.

**В: В каком случае `waitpid` не вернёт код завершения?**
О: Если процесс был убит сигналом (а не вышел через `exit`). `WIFEXITED(status)` будет `false`, `WIFSIGNALED(status)` — `true`, `WTERMSIG(status)` — номер убившего сигнала.

**В: Чем `fork + exec` отличается от `clone`?**
О: `fork + exec` — стандартная идиома: `fork` копирует процесс (COW), `exec` заменяет образ. `clone` — низкоуровневый syscall с гибкими флагами: можно разделить адресное пространство (`CLONE_VM`), файловые дескрипторы (`CLONE_FILES`) и т.д. `pthread_create` использует `clone` с разделением VM для создания потоков внутри одного процесса.

## Тема 6. Межпроцессное взаимодействие (IPC)

### Глоссарий

| Термин | Краткое определение |
|--------|---------------------|
| **IPC (Inter-Process Communication)** | Механизмы обмена данными и синхронизации между процессами |
| **`mmap(MAP_SHARED)`** | Общая память: два процесса видят один физический регион |
| **Race condition** | Гонка данных: результат зависит от порядка исполнения двух процессов без синхронизации |
| **Pipe (канал)** | Однонаправленный буфер в ядре; один конец для записи (`write-fd`), другой — для чтения (`read-fd`) |
| **`pipe(2)` / `pipe2(2)`** | Syscall: создаёт пару файловых дескрипторов — концы канала |
| **Сигнал (signal)** | Асинхронное уведомление процессу; `SIGINT`, `SIGTERM`, `SIGUSR1`, `SIGUSR2` и др. |
| **`kill(2)` / `raise(3)`** | Syscall/функция отправки сигнала процессу |
| **`sigaction(2)`** | Syscall: регистрирует обработчик сигнала |
| **`pause(2)` / `sigsuspend(2)`** | Syscall: приостанавливает процесс до получения сигнала |
| **Unix Domain Socket** | Сокет для IPC внутри одного хоста; адресуется путём в ФС (`/tmp/myapp.sock`) |
| **`SOCK_STREAM` / `SOCK_DGRAM`** | TCP-подобный (надёжный) / UDP-подобный (датаграммный) тип сокета |
| **Атомарность** | Операция либо выполняется полностью, либо не выполняется вовсе |

### Подробное объяснение

**Разделяемая память** (`mmap(MAP_SHARED)`) — самый быстрый IPC: данные передаются без копий через ядро, но требует явной синхронизации. Без синхронизации возникают **гонки данных**: процессы читают и пишут в произвольном порядке, и читатель может увидеть частично записанные данные (например, половину числа `unsigned int`).
**Сигналы** можно использовать для примитивной синхронизации: писатель заполнил буфер → посылает `SIGUSR1` читателю → читатель обрабатывает → посылает `SIGUSR2` писателю. Но сигнальный IPC ненадёжен при высоких нагрузках и плохо масштабируется. Правильный паттерн: обработчик сигнала только выставляет флаг `volatile sig_atomic_t`, а основной цикл проверяет флаг.
Для корректного ожидания сигнала используют `sigsuspend(&mask)`, который атомарно устанавливает маску сигналов и приостанавливает процесс — это исключает состояние гонки между проверкой флага и `pause()`.
**Pipe** — буфер в ядре (обычно 64 КБ). Запись `write(fd[1], ...)` и чтение `read(fd[0], ...)` **атомарны** для порций ≤ `PIPE_BUF` (4096 байт на Linux). Именно атомарность записи гарантирует, что читатель всегда видит цельные числа — в отличие от разделяемой памяти. Ядро также автоматически синхронизирует producer/consumer через блокировку: `read` блокируется, если нет данных; `write` — если буфер полон.
Чтобы перенаправить stdout родителя на stdin дочернего: создать `pipe`, в дочернем `dup2(fd[0], STDIN_FILENO)`, в родительском `dup2(fd[1], STDOUT_FILENO)`, закрыть оба конца `fd`.
**Unix Domain Sockets** (UDS) мощнее pipe: поддерживают двунаправленную связь, несколько клиентов одновременно, передачу файловых дескрипторов (`SCM_RIGHTS`) и учётных данных (`SCM_CREDENTIALS`). Эхо-сервер: `socket(AF_UNIX, SOCK_STREAM, 0)` → `bind(path)` → `listen` → цикл `accept` → `recv`/`send`.
Для обслуживания нескольких клиентов одновременно сервер использует `fork` на каждый `accept` (классика) или `select`/`epoll` (мультиплексирование I/O).
**`SIGINT`** (Ctrl+C) — стандартный сигнал завершения из терминала. Правильная обработка: установить обработчик через `sigaction`, в обработчике выставить флаг `shutdown_requested`, в основном цикле проверить флаг и освободить ресурсы перед выходом. Это гарантирует корректное завершение обоих процессов и освобождение общей памяти (`munmap`).
Порядок важен: `mmap`/`pipe`/сокет создаётся **до** `fork`, иначе дочерний процесс не унаследует ресурс.

### Вопросы и ответы

**В: Почему при передаче через pipe не нужна дополнительная синхронизация, а при разделяемой памяти нужна?**
О: Pipe — это очередь FIFO в ядре с встроенной синхронизацией: `read` блокируется на пустом, `write` — на полном. Записи ≤ PIPE_BUF атомарны. Разделяемая память — просто регион RAM: ядро не следит за порядком обращений, и процессы могут читать данные в момент, когда запись ещё не завершена.

**В: Что такое race condition при работе с разделяемой памятью? Приведите пример.**
О: Писатель пишет `unsigned int` (4 байта) и успевает записать 2 байта, когда ОС переключает контекст на читателя. Читатель видит число, в котором 2 старших байта — новые, 2 младших — старые, то есть мусор. Последовательность `{0, 1, 2, ...}` нарушается.

**В: Как правильно завершить пару процессов при получении SIGINT любым из них?**
О: Установить обработчик `SIGINT` в обоих процессах. Процесс, получивший сигнал, посылает `kill(other_pid, SIGTERM)`, выводит сообщение и завершается. Второй процесс, получив SIGTERM (или узнав что партнёр упал через `SIGPIPE`/`EPIPE`), выводит своё сообщение и завершается. Оба освобождают ресурсы в обработчиках или через `atexit`.

**В: Как передать файловый дескриптор между процессами через Unix Domain Socket?**
О: Используется вспомогательный канал `SCM_RIGHTS` через `sendmsg`/`recvmsg` с `struct cmsghdr`. Ядро дублирует дескриптор и создаёт новый fd в процессе-получателе, указывающий на тот же `struct file`. Это мощный механизм, используемый в systemd, Chrome sandbox и других.

## Тема 7. Пользователи и управление правами. SUID

### Глоссарий

| Термин | Краткое определение |
|--------|---------------------|
| **UID / GID** | User ID / Group ID — числовые идентификаторы пользователя и группы |
| **RUID (Real UID)** | Реальный UID: кто запустил процесс |
| **EUID (Effective UID)** | Эффективный UID: используется ядром для проверки прав доступа |
| **SUID-бит (Set-User-ID)** | Бит в inode исполняемого файла: при запуске EUID процесса = UID **владельца файла** |
| **SGID-бит** | Аналог SUID для групп: EGID = GID владельца файла |
| **`getuid()` / `geteuid()`** | Функции: получить RUID / EUID текущего процесса |
| **`setuid(2)` / `seteuid(2)`** | Syscall'ы: изменить UID/EUID (требуют прав или `CAP_SETUID`) |
| **`chmod u+s`** | Команда установки SUID-бита на файл |
| **DAC (Discretionary Access Control)** | Стандартная модель прав UNIX: владелец файла управляет правами |
| **`sudo` / `su`** | Утилиты для выполнения команд с правами другого пользователя (используют SUID) |
| **Capabilities** | Гранулярные привилегии Linux: замена монолитного root на отдельные способности (`CAP_NET_BIND_SERVICE`, `CAP_SYS_PTRACE` и др.) |
| **`passwd`, `/etc/shadow`** | `passwd` — SUID-программа; `/etc/shadow` — файл хэшей паролей, читаемый только root |

### Подробное объяснение

В UNIX-системах права доступа к файлу проверяются ядром по **EUID и EGID** процесса, а не по тому, кто его запустил. Обычно RUID == EUID, но механизм SUID позволяет это изменить.
Когда исполняемый файл имеет **SUID-бит** (`chmod u+s`) и владельцем является `root`, любой пользователь запускающий его получает EUID=0 на время работы программы. Так работают `passwd`, `sudo`, `ping` — они нуждаются в привилегированных операциях (запись в `/etc/shadow`, отправка raw-сокетов), но должны быть доступны обычным пользователям.
`getuid()` всегда возвращает RUID — кто реально запустил процесс. `geteuid()` возвращает EUID — с чьими правами работает. В SUID-программе: `getuid()` == UID пользователя, `geteuid()` == UID владельца файла (например, 0 для root).
Права файла хранятся в inode как 12 бит: три тройки `rwx` для owner/group/others + SUID/SGID/sticky. При проверке ядро: если `EUID == uid файла` → применяет owner-биты; если `EGID == gid файла` → group-биты; иначе — other-биты. `root` (EUID=0) проходит любые DAC-проверки (кроме execute).
**Sticky-бит** на директории (`/tmp`) означает: только владелец файла может удалить или переименовать его в этой директории, даже если у других есть `w` на директорию.
**SUID-безопасность**: SUID-программа расширяет поверхность атаки — любой баг в ней может дать атакующему root. Поэтому SUID-бинарников должно быть минимум, они должны быть максимально маленькими и сразу отбрасывать привилегии (`seteuid(getuid())`) после выполнения привилегированной части.
**Linux Capabilities** — современная замена монолитному root: вместо EUID=0 программе даётся конкретная способность, например `CAP_NET_BIND_SERVICE` (биндинг порта <1024) или `CAP_DAC_OVERRIDE` (обход DAC-проверок). `ping` на современных системах использует capabilities вместо SUID.
`/proc/pid/status` показывает строки `Uid: RUID EUID SUID FSUID` и `Gid:` — можно наблюдать переключение EUID в реальном времени.
Понимание SUID критично для системного администрирования и безопасности: `find / -perm -4000 2>/dev/null` — команда поиска всех SUID-файлов в системе, один из первых шагов при аудите безопасности или privilege escalation.

### Вопросы и ответы

**В: Чем RUID отличается от EUID? Когда они различаются?**
О: RUID — кто запустил процесс (наследуется при `fork`, не меняется при `exec`). EUID — с чьими правами работает (используется ядром для проверки доступа к файлам и syscall'ам). Они различаются при запуске SUID-программы: `passwd` запускается обычным пользователем (RUID=1000), но имеет SUID root, значит EUID=0. Также `setuid`/`seteuid` могут их разделить.

**В: Как работает `passwd` — почему обычный пользователь может менять свой пароль, если `/etc/shadow` читает только root?**
О: Бинарник `/usr/bin/passwd` имеет SUID-бит и владелец — root. При запуске любым пользователем EUID становится 0. С EUID=0 программа открывает `/etc/shadow`, проверяет старый пароль и записывает новый хэш. После этого сбрасывает привилегии. Именно SUID позволяет реализовать этот контролируемый доступ.

**В: Почему SUID на скриптах (shell-скриптах) не работает в Linux?**
О: По соображениям безопасности ядро Linux игнорирует SUID-бит на интерпретируемых файлах (`#!/bin/bash`). SUID работает только на ELF-бинарниках. Это сделано потому, что скрипт запускает оболочку, которая до выполнения скрипта может читать конфиг-файлы пользователя (`~/.bashrc`), что создаёт огромную поверхность атаки.

**В: Что такое privilege escalation и как SUID участвует в атаках?**
О: Privilege escalation — получение прав выше тех, что положены пользователю. Если SUID-программа имеет уязвимость (buffer overflow, path traversal, race condition), атакующий может выполнить произвольный код с EUID=0. Классика: уязвимый SUID-бинарник с `system()`, которому можно подсунуть вредоносный `PATH`. Именно поэтому количество SUID-файлов минимизируют, а современные системы используют capabilities.
